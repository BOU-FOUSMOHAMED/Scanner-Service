## 1. Architecture Overview

The **Scanner Service** is a microservice within the Web Security Assessment Platform responsible for the lifecycle management of security scan targets and their associated results. It follows a **Layered Architecture** with an **Event-Driven extension** for asynchronous scanning orchestration.

### 3.2 Module Responsibility Matrix

| Module | Responsibility | Key Rules |
|---|---|---|
| **Controller** | HTTP entry-point, request deserialization, response serialization, HTTP status mapping | `@Valid` on request bodies; returns `ResponseEntity<T>`; delegates to service |
| **Service (Interface)** | Defines the service contract вҖ” what operations are available | No implementation details; `@Transactional` boundary defined at impl |
| **Service (Impl)** | Business logic, validation orchestration, transactional boundaries | `@Transactional` on class; reads use `readOnly=true`; throws `BusinessException` or `ResourceNotFoundException` |
| **Repository** | Data access via Spring Data JPA; custom query derivation | Extends `JpaRepository<Entity, UUID>`; method naming drives queries |
| **Entity** | JPA mapping, lifecycle callbacks (`@PrePersist`, `@PreUpdate`) | UUID PK via `GenerationType.UUID`; `@Enumerated(STRING)` for enums |
| **Mapper** | Entity-to-DTO transformation | Static utility class; no framework dependency; single `toResponse()` method |
| **DTO (Request)** | Input validation contracts via Bean Validation | Records with `@NotBlank`, `@Pattern`, `@DecimalMin/Max` |
| **DTO (Response)** | Output contract вҖ” what clients receive | Records; no business logic |
| **Exception Handler** | Centralized error handling for all controllers | `@RestControllerAdvice`; maps exceptions to `ErrorResponse` with status, message, field errors |
| **UrlValidator** | URL normalization (add https:// if missing) + validation (regex + URI parsing) | Max length 2048; protocol must be http/https/ftp; host must contain `.` |
| **DataInitializer** | Seeds demo data on first run (conditional on empty tables) | `CommandLineRunner`; only runs when both tables are empty |

### 13.2 Hardening Recommendations

1. **Externalize secrets**: Move DB credentials to environment variables or a vault.
2. **Add Spring Security**: At minimum, enable management endpoint security.
3. **CORS policy**: Restrict origins if the API is consumed by browser clients.
4. **Rate limiting**: Add at gateway level or via Spring Cloud Gateway filter.
5. **Audit logging**: Log all mutating operations (create, update, delete).

## 14. Architecture Decisions (ADRs)

### ADR-001: Layered Architecture over Hexagonal

| Attribute | Value |
|---|---|
| **Decision** | Use traditional layered architecture (Controller вҶ’ Service вҶ’ Repository) |
| **Rationale** | The service has a single domain (scan management) with no port/adapter complexity. Hexagonal architecture would introduce unnecessary indirection. |
| **Trade-off** | Direct coupling between layers; testing requires mocking at each boundary. |

### ADR-002: Static Mappers over MapStruct

| Attribute | Value |
|---|---|
| **Decision** | Hand-written static `Mapper` classes (no MapStruct dependency) |
| **Rationale** | Only two mappers with trivial field mappings; MapStruct would add compile-time annotation processing overhead with marginal benefit. |
| **Trade-off** | Manual maintenance; no performance penalty. For 2 mappers, acceptable. |

### ADR-003: UUID Primary Keys over Auto-Increment

| Attribute | Value |
|---|---|
| **Decision** | `GenerationType.UUID` for all entity IDs |
| **Rationale** | Prevents ID enumeration attacks; supports distributed database sharding; aligns with event-driven correlation IDs. |
| **Trade-off** | Larger index size; slightly slower inserts vs `SERIAL`. |

### ADR-004: Logical Relationship by URL (No FK)

| Attribute | Value |
|---|---|
| **Decision** | `scan_results.target` references `scan_targets.url` by value, not by foreign key |
| **Rationale** | Results can arrive from external scanners (via RabbitMQ) for targets that do not yet exist in the database. A physical FK would break this decoupling. |
| **Trade-off** | No referential integrity enforcement at database level; application must handle orphans. |

### ADR-005: DTO Records over Classes

| Attribute | Value |
|---|---|
| **Decision** | Java `record` types for all DTOs |
| **Rationale** | Immutability, `equals()`/`hashCode()`/`toString()` out of the box, concise syntax. |
| **Trade-off** | No JPA/Jackson proxies; Jackson serialization works natively with records in Spring Boot 3. |

### ADR-006: Enum as String in Database

| Attribute | Value |
|---|---|
| **Decision** | `@Enumerated(EnumType.STRING)` for `ScanStatus` |
| **Rationale** | Readable in DB queries; allows adding new values without migration; no DB-specific enum type. |
| **Trade-off** | Slightly more storage than ordinal; case-sensitive matching. |
